1. Project Overview

🏥 Purpose

This desktop application digitises the appointment and queue management process of a hospital. It provides three role-based dashboards — Admin, Doctor, and Patient — each with context-specific features. The goal is to eliminate paper-based queues, manage doctors' schedules, and give patients a real-time view of their appointment status.

🛠️ Technology Stack

  • Frontend (UI): JavaFX 21 with FXML layout files
  • Styling: JavaFX CSS (style.css) — inspired by Tailwind/modern card design
  • Backend Logic: Java 17 — MVC architecture
  • Database: PostgreSQL hosted on Supabase (cloud) — accessed via JDBC
  • Build System: Apache Maven 3.9.6 (bundled) + pom.xml
  • ORM: Raw JDBC (no Hibernate/JPA)

3. Architecture & Design Pattern

🏗️ Model-View-Controller (MVC)

The project follows a clean MVC layered architecture:


LayerPackageResponsibility
Modelcom.hospital.modelsPure data objects — User, Doctor, Patient, Appointment. No logic, no UI.
Viewresources/views/*.fxml
resources/styles/style.css		FXML defines UI layout declaratively. CSS controls visual appearance.
Controllercom.hospital.controllersHandles user events (clicks), fetches data via DAO, updates the View.
DAOcom.hospital.daoData Access Objects — all SQL queries. Decoupled from UI completely.
Servicecom.hospital.servicesBusiness logic — queue ordering, conflict detection, sorted in-memory lists.
Utilcom.hospital.utilsCross-cutting concerns — DB connection, session state, custom exceptions.

Why MVC? It separates UI concerns from data logic. A developer can swap PostgreSQL for MySQL/SQLite purely in the DAO layer without touching controllers or FXML views. Similarly, UI redesigns don't break business logic.

4. Component Deep-Dive

4.1 Entry Point

📄 Main.java — com.hospital.main

  • Extends javafx.application.Application — the root of every JavaFX app.
  • start(Stage): Resolves the URL for Login.fxml, calls FXMLLoader.load(), creates a Scene(root, 900, 600), sets the window title "Smart Hospital Appointment & Queue Management", then shows the stage.
  • main(): Calls launch(args) — required JavaFX bootstrap.
  • Error handling: prints a message if Login.fxml cannot be found.

4.2 Models

FileFieldsSignificance
User.java id, name, username, password, role Abstract base class. Cannot be instantiated directly. Doctor and Patient both inherit from it. Role field ("Admin"/"Doctor"/"Patient") drives routing logic post-login.
Doctor.java +specialization, +consultationFee Extends User. Used to populate AdminController's doctor list and PatientController's doctor picker ComboBox.
Patient.java +bloodGroup, +contactNumber, +age Extends User. Medical identity object linked to every Appointment. Displayed in DoctorController's patient list.
Appointment.java appointmentId, Patient, Doctor, date, time, status Core domain object. Links a Patient to a Doctor at a specific date/time slot. Status can be "Pending"/"Confirmed"/"Completed".

4.3 Data Access Objects (DAO)

FileKey MethodsWhat It Does
UserDAO.java authenticate(), insertUser() authenticate(username, password): Runs a SELECT on the Users table; if match found, checks role and returns a typed Doctor/Patient/Admin object.
insertUser(User): Inserts into Users table + the relevant sub-table (DoctorDetails or PatientDetails) using RETURN_GENERATED_KEYS to link records.
DoctorDAO.java getAllDoctors(), getDoctorById() JOINs Users + DoctorDetails tables to build Doctor objects with specialization and fee. Used by Admin view and Patient booking form.
PatientDAO.java getAllPatients(), getPatientById() JOINs Users + PatientDetails to build full Patient objects. Used by Admin and Doctor dashboards to list patients.
AppointmentDAO.java bookAppointment(), getAppointmentsByPatient(), getAppointmentsByDoctor(), getAllAppointments() bookAppointment(): INSERTs a new appointment record, returns boolean success.
getAppointmentsByDoctor(): Used for doctor's live queue — fetches today's appointments for the logged-in doctor.
All queries return typed objects (not raw result sets), keeping controllers clean.

4.4 Services (Business Logic)

⚙️ QueueService.java

  • Manages a daily in-memory appointment list (ArrayList).
  • addAppointment(Appointment): Checks if an identical slot exists; throws SlotConflictException if it does. Otherwise appends.
  • getPosition(Appointment): Iterates the queue and returns 1-based position index of the appointment (for status display).
  • Purpose: Acts as an optimistic in-memory buffer so the UI doesn't need to query DB for every queue position update.

⚙️ DataStorageService.java

  • Holds sorted in-memory collections: ArrayList<Doctor>, ArrayList<Patient>, ArrayList<Appointment>.
  • loadDoctors(), loadPatients(): Populate lists from DAO calls.
  • Appointments are sorted by time using Collections.sort(... Comparator.comparing(Appointment::getTime)).
  • Purpose: Caches DB data locally for the session — reduces repeated DB round-trips for the same list data.

4.5 Controllers

🖥️ LoginController.java

  • Bound to Login.fxml — reads usernameField + passwordField on button click.
  • Calls UserDAO.authenticate(). On success: stores user in Session.setLoggedInUser().
  • Role-based routing: loads AdminDashboard.fxml, DoctorDashboard.fxml, or PatientBooking.fxml based on user.getRole().
  • On failure: shows red error message in errorLabel.
  • Catches SQLException separately to inform user if DB is unreachable.

🖥️ AdminController.java

  • Provides three navigation options: View Doctors, View Patients, View Appointments.
  • Each view dynamically builds a TableView inside cardArea (no separate FXML per sub-view).
  • showDoctors(): Fetches all doctors via DoctorDAO; populates table with Name, Specialization, Fee columns.
  • showPatients(): Fetches all patients; shows Name, Blood Group, Age, Contact.
  • showAppointments(): Fetches all appointments; shows Patient Name, Doctor Name, Date, Time, Status.
  • Uses SimpleStringProperty for TableColumn cell value factories.

🖥️ DoctorController.java

  • Most complex controller — manages a background thread for live queue polling.
  • showQueue(): Launches a Thread (liveQueueThread) that polls the DB every 5 seconds for today's appointments for the logged-in doctor. Updates the UI using Platform.runLater() (thread-safe JavaFX update).
  • showSchedule(): Displays upcoming appointments in a table (sets isViewingSchedule = true).
  • showPatients(): Lists all patients linked to this doctor's appointments.
  • stopThread(): Sets running = false and calls liveQueueThread.interrupt() — called when navigating away to prevent memory leaks.
  • Header label updates dynamically to reflect current view ("My Queue", "My Schedule", "My Patients").

🖥️ PatientController.java

  • Loads to PatientBooking.fxml and populates a ComboBox of doctors from DoctorDAO.
  • showBooking(): Patient picks Doctor → Date (DatePicker) → Time Slot (ComboBox) → clicks "Book Appointment".
  • On booking: calls AppointmentDAO.bookAppointment(); on success shows confirmation card.
  • showAppointments(): Fetches appointments for Session.getLoggedInUser() and renders a VBox of appointment cards with date, doctor name, time, and status.
  • Uses VBox.setVgrow(table, Priority.ALWAYS) for responsive layouts.

4.6 Utilities

FilePurposeKey Detail
DatabaseConnection.java JDBC Connection factory Single static getConnection() method. URL points to Supabase PostgreSQL cluster using the postgresql JDBC driver. Credentials are hardcoded (not ideal for production).
DatabaseSetup.java One-time DB seeder Has its own main() so it can be run independently. Executes batch inserts of sample Admin, Doctor, and Patient accounts into the live DB. Friendly error messages guide setup.
Session.java Global user session Static singleton. setLoggedInUser(User), getLoggedInUser(), clear(). Allows any controller to access the currently authenticated user without passing it through constructors.
SlotConflictException.java Custom checked exception Thrown by QueueService.addAppointment() when a duplicate time slot is detected. Extends Exception with a message constructor.

5. Application Workflow

5.1 Login & Role Routing

1

App Launch → Login Screen

Main.java loads Login.fxml into a 900×600 Stage. User sees username/password fields.

2

Credential Submission

LoginController reads fields → calls UserDAO.authenticate() → SQL SELECT on Users table.

3

Session Storage

Authenticated User object (Doctor/Patient/Admin) stored in Session.setLoggedInUser().

4

Role-Based Navigation

Role = "Admin" → AdminDashboard.fxml | "Doctor" → DoctorDashboard.fxml | "Patient" → PatientBooking.fxml

5.2 Patient Books an Appointment

1

Patient Dashboard Loads

PatientController.initialize() → DoctorDAO.getAllDoctors() → populates doctor ComboBox.

2

Selections Made

Patient picks Doctor, Date (DatePicker), Time Slot (ComboBox with hardcoded slots e.g. "09:00", "10:00").

3

Book Appointment Button

Creates Appointment object → calls AppointmentDAO.bookAppointment() → SQL INSERT into Appointments table.

4

Conflict Check + Confirmation

QueueService checks if slot is free. If OK → success card shown. If conflict → SlotConflictException message shown to user.

5.3 Doctor's Real-Time Queue

1

Doctor Logs In → Clicks "View Queue"

DoctorController.showQueue() is triggered.

2

Background Thread Spawned

A new Thread runs in a loop: sleeps 5s → calls AppointmentDAO.getAppointmentsByDoctor() → updates TableView via Platform.runLater().

3

Live UI Updates

Table refreshes every 5 seconds showing patients in queue order by appointment time.

4

Thread Cleanup

stopThread() called on navigation (running=false + interrupt) — prevents zombie threads.

6. Database Design

TablePrimary KeyKey ColumnsPurpose
Users id (SERIAL) name, username (UNIQUE), password, role Master identity table — all three roles share this. Role = 'Admin'|'Doctor'|'Patient'.
DoctorDetails doctor_id (FK → Users.id) specialization, consultation_fee Doctor-specific extended data. Linked 1:1 with Users via ON DELETE CASCADE.
PatientDetails patient_id (FK → Users.id) blood_group, contact_number, age Patient-specific medical data. ON DELETE CASCADE from Users.
Appointments id (SERIAL) patient_id (FK), doctor_id (FK), appointment_date, appointment_time, status Core booking record. Joins patient and doctor. Queried by date to build today's queue.

Database is hosted on Supabase (cloud PostgreSQL). This means no local PostgreSQL installation is needed. The JDBC URL in DatabaseConnection.java points directly to the Supabase cluster endpoint. All team members connect to the same live database automatically.

schema.sql creates the four tables from scratch. seed.sql inserts TRUNCATE + sample rows for testing: 1 Admin, 2+ Doctors, 3+ Patients, 5+ Appointments spanning multiple dates.

8. What Remains to Build

🔴 High Priority (Core Features)

  1. Logout functionality — Every dashboard needs a Logout button that calls Session.clear() and returns to Login screen.
  2. Appointment Status Update — Doctor should be able to mark "Pending" appointments as "Confirmed" or "Completed" from the queue. Requires an UPDATE SQL statement in AppointmentDAO.
  3. Admin CRUD for Doctors/Patients — Admin dashboard currently only reads data. Add/Edit/Remove functionality needs forms + UserDAO.insertUser() wiring.
  4. Input Validation — Login fields should not submit when empty. Date picker should not allow past dates. SlotConflictException feedback needs to be surfaced to the user as a dialog, not just a console error.

🟡 Medium Priority (Important but Not Blocking)

  1. Patient Registration / Sign-Up Screen — Currently patients must be seeded into the DB. A self-registration FXML form + UserDAO.insertUser() flow is needed.
  2. Password Hashing — Passwords are stored as plain text. BCrypt or SHA-256 hashing should be added in UserDAO before production use.
  3. QueueService ↔ DB Integration — QueueService currently manages an in-memory list. It should be initialised from today's DB appointments on startup, and stay in sync with AppointmentDAO.
  4. Doctor schedule full view — DoctorController.showSchedule() fetches data, but the TableView column configuration needs completing for all fields.

🟢 Low Priority (Polish)

  1. Error handling improvement — Replace all e.printStackTrace() calls with JavaFX Alert dialogs.
  2. Loading spinners — DB calls on the UI thread can freeze the window. Long queries should run in a Task<Void> with a ProgressIndicator.
  3. Unit Tests — DAO methods and QueueService logic should have JUnit 5 tests for correctness.
  4. Externalise DB credentials — Move URL/USER/PASSWORD from DatabaseConnection.java into a .env or application.properties file.
  5. Appointment cancellation by patient — Patient should be able to cancel a pending appointment.

9. How to Run the Application

🚀 Prerequisites

  • Java 17 or later must be installed. Run: java -version
  • No PostgreSQL local install needed — DB is on Supabase cloud.
  • Maven is bundled inside apache-maven-3.9.6/ — no global install required.

▶️ Run the App

Windows (bundled Maven):

.\apache-maven-3.9.6\bin\mvn.cmd clean compile javafx:run

Windows (global Maven installed):

mvn clean compile javafx:run
